This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Commands

```bash
# Install all workspaces
npm install

# Dashboard dev server (port 3001)
cd apps/dashboard && npm run dev

# Build dashboard
npm run build                          # builds apps/dashboard
npm run build:shared                   # builds packages/shared (required before building services)
npm run build:auth-service             # builds a single service

# Lint backend services
npm run lint                           # services/*/src
npm run lint:fix
npm run lint:shared                    # packages/shared

# Database
npm run db:push                        # push Drizzle schema to DB
npm run db:studio                      # open Drizzle Studio

# Start all services + dashboard together
bash scripts/start-dev.sh

# E2E tests (from apps/dashboard/)
cd apps/dashboard && npm run test:e2e
cd apps/dashboard && npm run test:e2e:ui    # Playwright UI mode
cd apps/dashboard && npm run test:e2e:headed

# Individual service dev
cd services/auth-service && npm run dev    # tsx watch (hot-reload)
```

There are no unit tests configured at the root. The only automated test suite is Playwright E2E in `apps/dashboard`.

## Architecture

This is an npm workspaces monorepo:
- `apps/dashboard` — Next.js 15 frontend + API routes (App Router), runs on port 3001
- `apps/mobile` — Expo React Native app
- `packages/shared` — shared Drizzle schema, middleware, utilities; **must be built before services**
- `services/` — 9 Express microservices: auth, user, team, project, task, chat, file, invoice, notification (plus `_template`)
- `infrastructure/database/migrations/` — Drizzle-managed SQL migrations

### Database

Single Neon/PostgreSQL database. Schema is defined in `packages/shared/schema.ts` and used via Drizzle ORM. The `drizzle.config.ts` at the repo root points to that schema. **Exception**: `services/chat-service` has its own Drizzle config and dedicated database.

Each microservice uses a lazy-initialized connection pool (`services/<svc>/src/db/index.ts` pattern) with a Proxy wrapper to defer connection until first use — safe for serverless.

### Auth flow

JWT HS256 sessions stored in the `session` cookie. The shared secret (`JWT_SECRET`) is used by both the dashboard (`apps/dashboard/lib/auth.ts`) and microservices via `@project-scope-analyzer/shared`. Session validation happens in three places:
1. **Edge middleware** (`apps/dashboard/middleware.ts`) — guards all protected routes, verifies JWT, optionally caches userId in Upstash Redis with a circuit-breaker to survive Redis outages
2. **Next.js API routes** — call `getSession()` from `lib/auth.ts`
3. **Microservices** — call auth-service or verify JWT directly

The auth-service URL is proxied through `apps/dashboard/lib/auth-service.ts` using `AUTH_SERVICE_URL` env var.

### Next.js API route pattern

All API routes in `apps/dashboard/app/api/` follow this structure:
1. `getSession()` → 401 if missing
2. `withRateLimit()` for mutation routes
3. `cached(cacheKey, fn, { ttl })` helper from `lib/redis.ts` for reads
4. Business logic via `@/server/storage` (storage abstraction over the DB)

Cache keys are generated by `lib/cache-keys.ts` (`CacheKeys.*`) with `CacheTTL` constants. Invalidation helpers (`getProjectInvalidationKeys`, etc.) are also there.

### Real-time (Ably)

`apps/dashboard/lib/pusher-server.ts` exports `triggerChatEvent()` using Ably. The file has a `getPusherServer()` export marked `@deprecated` — use `getAblyServer()` directly. Client hooks are in `hooks/usePusher.ts` and `hooks/useChatPusher.ts`. Channel naming: `private-chat-{chatId}`, `presence-user-{userId}`.

### Client-side data fetching

TanStack Query (`@tanstack/react-query`) with a singleton `QueryClient` (`lib/queryClient.ts`). Default stale time is 60 s; cache is persisted to `localStorage` (saved on `beforeunload` and every 30 s). Use `useQueries.ts` hooks for data access; avoid fetching in components directly.

### i18n

`next-intl` with `en` and `uk` locales. Locale prefix is always present in the URL (`/en/...`, `/uk/...`). Locale is detected first from the `NEXT_LOCALE` cookie, then from the URL path. All page files live under `apps/dashboard/app/[locale]/`.

### Microservice structure

Every service (`services/<svc>/src/`) has:
- `app.ts` — Express app factory (`createApp()`)
- `index.ts` — entry point calling `startServer()`
- `config/index.ts` — env-based config (DATABASE_URL, REDIS_URL, etc.)
- `routes/` — route registration
- `controllers/` — request handlers
- `services/` — business logic
- `db/` — Drizzle instance + schema imports from `packages/shared`
- `middleware/` — rate-limiting, auth guards, etc.
- `swagger.ts` — OpenAPI spec; Swagger UI served at `/api-docs`

Service-to-service calls use API keys via `AUTH_SERVICE_API_KEY` (and similar per-service env vars), never JWTs.

### Deployment

- **Railway**: deploys `apps/dashboard` (`railway.json`)
- **Vercel**: deploys `apps/dashboard/app/api/**` as serverless functions (`vercel.json`) and each `services/<svc>` as a separate Vercel project

## Key env vars

| Var | Used by |
|-----|---------|
| `DATABASE_URL` | all services, dashboard, drizzle.config.ts |
| `JWT_SECRET` | dashboard middleware, auth-service, shared |
| `UPSTASH_REDIS_REST_URL` / `_TOKEN` | dashboard middleware (session cache), API routes |
| `ABLY_API_KEY` | dashboard pusher-server.ts |
| `AUTH_SERVICE_URL` | dashboard lib/auth-service.ts |
| `AUTH_SERVICE_API_KEY` | service-to-service auth calls |
